# 无垠博客写作

## 先判断文章类型

- **技术实践 / 排障**：从一个具体痛点、报错或高频摩擦开场；先给可运行的结论和配置，再解释原因、失败路径、边界与排查方法。保留准确的命令、配置、表格和代码块。
- **开源项目发布**：按“为什么做 → 解决什么摩擦 → 关键取舍与边界 → 技术方案 → 仓库与下一步”展开。不要把小工具夸成平台；讲清它不做什么。
- **技术观点**：先提出鲜明判断，再用真实的修改路径、接口、代码或工作流把判断落地；最后收束为可操作的提问法或原则。
- **随笔 / 技术哲思**：由一个可感知的场景或细节起笔，将技术概念延展到人的经验；用短段落和递进的小节推进，结尾回到具体行动、问题或带张力的意象，不做空泛鸡汤。
- **算法 / 教程**：先用一句话给直觉，再解释朴素做法为何不足，逐轮演示核心机制，最后提供实现、复杂度、适用范围和限制。

若用户未指定体裁，根据主题和已有材料选择最贴切的一种；不要为了模仿而硬塞哲学化表达。

## 写作流程

1. 先阅读与主题、体裁最相近的文章；中文看 `docs/posts/`，英文看 `docs/en/posts/`。优先借鉴结构、论证密度和信息组织，不复用原句。
2. 先列出文章的中心判断、读者已遇到的具体场景、证据或可验证的细节，以及明确的适用边界。事实、版本、价格、接口和命令需先核实；不确定就标出待确认，不要编造。
3. 先写能独立成立的摘要，再围绕摘要安排小节。每节只推进一个观点；长文用小标题、列表、表格或代码块降低认知负担。
4. 完稿后删除重复表态、泛泛的“革命性”“颠覆性”“赋能”等空词，并检查结论是否比开头多提供了可执行的理解或动作。

## 让文章对读者有用

- 明确这篇文章要帮助哪类读者理解、判断或完成什么；不必把博客写成销售页，但不要只写作者自己的功能清单或心情记录。
- 每提出一个功能、事实或观点，都追问一次“这对读者意味着什么”。用具体后果、使用场景、时间成本、配置差异或认知变化回答，而不是追加形容词。
- 清晰优先于机巧：先说结论，再补限定和背景；使用读者熟悉的词和主动语态。不要用行话、缩略语或长句来伪装深度。
- 具体优先于抽象：能写报错、命令、文件、版本、步骤、数字或可观察场景时，不写“更高效”“更现代”“体验更好”这类无支撑的结论。
- 让重要主张有相称的证据。可用复现步骤、代码、官方文档、数据、亲身观察或明确的推理；没有证据时，收窄为个人判断，不制造统计、引语或用户反馈。
- 预判读者合理的疑问：前提是否成立、适用于谁、有哪些限制、下一步怎么做。技术文章给出明确的可执行入口；随笔给出值得停留的具体问题或视角，而非生硬 CTA。

## 中文语气与节奏

- 使用第一人称的观察和判断，但让事实、例子和推理承担论证；语气直接、克制，不端着，也不堆网络热词。
- 需要澄清概念时，直接说明误解和判断。不要反复依赖固定的对照句式。
- 从具体摩擦切入：一次报错、一个多余步骤、一段遗留代码、一个日常习惯。随后逐层上升，不跳过中间推理。
- 对工程取舍保持明确：小工具也要可靠；错误状态要说清；默认行为要克制；边界、隐私和失败路径必须写出来。
- 随笔允许较强的画面与比喻，但让技术事实和情绪各有位置。用有分量的短句收束段落，避免每段都追求“金句”。

## 英文写法

- 英文稿是重写，不是逐句翻译。保留事实、链接、代码、章节意图与论证顺序，改成自然、清晰的英文表达。
- 按内容决定句子的长短，优先普通词汇和具体动词。技术文章直接说明结论，不套用固定的开头或收束句，也不要硬译中文修辞。
- 延续中文稿的克制感，但减少夸饰。随笔可保留意象与节奏，技术文章优先准确、可扫描和可复现。
- 专有名词、项目名、命令、代码、URL 和产品界面名称保持原样；不要把不稳定的术语强行意译。

## 润色既有文章

以保留原有中心判断、事实与作者声音为前提，分轮检查，不要为“更像 AI”而重写。

1. **清晰**：补足指代与上下文，拆开承载太多信息的句子，保证一段只推进一个主题。
2. **语气**：统一直接、克制、可信的口吻；删除突然出现的企业话术、过度承诺或不合语境的玩笑。
3. **读者价值**：为只罗列功能、概念或经历的段落补上“所以呢”——它改变了什么判断、减少了什么摩擦、带来了什么实际后果。
4. **证据与具体性**：把笼统主张换成可核实的例子、条件或边界；无法证明的绝对化表述要删去或降格为观点。
5. **节奏**：删掉填充词和重复结论，前置重点，混合短句与必要的长句；保留段落间的逻辑过渡和阅读留白。

每轮修改后回看前一轮：润色不能改变原意、引入新事实，或损害双语稿之间的一致性。向用户交付润色稿时，简要说明实质性调整及其理由；不要只给笼统的“已优化”。

## 站点成稿格式

- 默认同时创建或更新 `docs/posts/<year>/<slug>.md` 与 `docs/en/posts/<year>/<slug>.md`，文件名相同；用户明确要求单语时除外。
- 两篇均使用 `title`、`date`、`tags`、`description`、`outline: deep`、`aside: true` 六项 frontmatter；紧接与 title 一致的一级标题。
- 在两个独立的 `<!-- DESC SEP -->` 标记之间写首页摘要。摘要与 `description` 同义但不机械重复，应能单独说明文章主张或读者收益。
- 中英文标签语义对齐；内部链接使用站点路径，不写 `.html`。图片写有意义的替代文本。
- 发布工具或方案时，保留读者真正需要的命令、完整配置、已知限制和排查步骤；代码和配置必须与文字中的前提一致。

## 最终自检

- 开头是否尽快给出主题与问题，而不是从宽泛背景开始？
- 每个具体结论是否有例子、配置、机制或清晰推理支撑？
- 每个功能、概念或个人观察，是否说明了它对目标读者的实际意义？
- 是否使用了具体、主动、可理解的表达，并删掉了无证据的夸张和企业话术？
- 是否明确了不支持的范围、风险或替代方案，而非用含糊兜底掩盖问题？
- 双语文章是否各自读起来自然，且事实、链接和结论一致？
- 文章是否符合仓库的 Markdown、frontmatter 和双语同步约定？
