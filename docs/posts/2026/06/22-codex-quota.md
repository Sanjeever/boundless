# Codex Quota

<!-- DESC SEP -->

我开源了 Codex Quota，一个用于在本地菜单栏或系统托盘查看 Codex 用量的小工具。

<!-- DESC SEP -->

## 为什么做这个项目？

最近 Codex 已经越来越像一个会长期挂在日常开发流程里的工具。

但只要一个工具进入日常使用，用量就会从“偶尔看一眼”的信息，变成一种需要持续感知的状态。尤其是当你正在连续处理几个任务时，最不希望发生的事情就是：写到一半才发现 5 小时额度快没了，或者周额度已经进入危险区。

官方页面当然能看这些信息，但每次都打开浏览器、进入设置页、等页面加载，再回到编辑器，这个路径太长了。

我想要的是一个更低摩擦的东西：它不打扰我，但我需要时一眼就能看到。

于是有了 **Codex Quota**。

## 它解决的不是“看数据”，而是“少切上下文”

Codex Quota 是一个非官方的本地 Tauri 桌面应用。

在 macOS 上，它会显示在菜单栏里，标题类似：

```text
Codex 5h 96% | Weekly 36%
```

在 Windows 上，它会显示为系统托盘图标，通过 tooltip 和托盘菜单展示当前状态。

它关注的是几个最直接的问题：

- 5 小时窗口还剩多少额度？
- 周额度还剩多少？
- 什么时候重置？
- 最近一次刷新是否成功？
- 当前显示的是不是过期数据？

这些信息并不复杂，但如果每次都要打开页面查看，就会变成反复发生的小摩擦。

很多工具的价值并不在于“提供了一个全新的能力”，而在于把一个高频动作从三四步压缩到一步。

Codex Quota 做的就是这件事。

## 为什么不用 API key？

这个项目刻意没有走 API key 路线。

原因很简单：我不想让一个查看用量的小工具变成另一个需要管理密钥、粘贴 token、复制 curl 命令的工具。

Codex Quota 不要求你输入 API key，不要求你粘贴 cookie，也不要求你提供 bearer token。它会打开一个本地 Tauri WebView，让你在 ChatGPT 页面里正常登录，然后在这个已经认证的页面运行时中请求用量接口。

换句话说，它复用的是你本机 WebView 里的 ChatGPT 登录态，而不是让你把凭据交给应用。

这里有一个很重要的边界：访问令牌只在当前 WebView 页面运行时里用于这一次请求，不返回给 Rust 侧，不写入本地存储，不显示在 Debug 窗口里，也不记录日志。这不意味着 WebView 登录态没有风险；安装和运行任何能访问已登录页面的本地应用前，仍应审查代码来源与发布包。

我希望这个工具足够透明，也足够克制。

它只是把已经能在网页里看到的信息，用更适合桌面使用的方式展示出来。

## 本地优先，而不是账号中台

Codex Quota 没有服务端。

它不会把你的用量上传到某个后台，也不会做账号同步，更不会尝试做“团队用量面板”之类的功能。

当前版本保存的只是一些本地状态：

- 刷新间隔
- 是否开机启动
- 最近一次成功的用量快照
- 最近更新时间

这些信息保存在 Tauri 应用自己的本地数据目录里。

如果刷新失败，但之前有成功快照，托盘仍然会展示旧数据，并明确标记为 stale。这样做的目的不是假装数据还新，而是在网络异常、接口错误或登录态过期时，仍然保留一个可参考的最后状态。

这个设计很朴素：能准确就准确，不能准确就明确告诉你哪里不准确。

不要猜，也不要静默伪装。

## 错误应该清楚，而不是“看起来正常”

Codex Quota 使用的是 ChatGPT Web 里的内部接口。

这意味着它天然有两个限制：接口可能变化，且它不是官方提供或保证的用量读取方式。

所以我没有让它在解析失败时去页面上扒文本，也没有做一堆看似聪明的兜底逻辑。如果接口返回异常、JSON 结构变化、登录态失效或网络断开，应用会进入明确的错误状态，例如：

- `Auth required`
- `Request timeout`
- `Offline`
- `API error`
- `Parse error`

这比展示一个可能错误的额度更重要。

对于这种工具来说，错不一定可怕，可怕的是错了还装作自己没错。

如果依赖的内部接口变了，正确的行为应该是让问题暴露出来，然后修代码，而不是用猜测结果继续误导用户。

## 调试窗口只服务于排查问题

除了托盘菜单，Codex Quota 还有一个 Debug details 窗口。

这个窗口主要用于排查问题，会显示本地运行状态、账号信息、两个额度窗口、重置时间、最近错误以及脱敏后的 JSON。

这里也有一个边界：窗口里可以看到用于本机排查的账号信息，但复制 JSON 时会做脱敏处理。`user_id`、`account_id` 会被替换为 `[redacted]`，邮箱也会被遮挡。

分享问题时，应该优先使用 `Copy summary` 或脱敏后的 `Copy JSON`。

不要把 token、cookie、原始请求头或其他凭据贴到 Issue 里。

这是小工具，但不代表可以在凭据处理上随便。

## 为什么换成 Tauri？

Codex Quota 现在是 Tauri v2 应用。

这个版本不是旧 Electron 数据的延续，也不会读取、转换或删除旧 Electron 会话、cookie、缓存或配置。第一次启动 Tauri 版本时，需要重新登录 ChatGPT。

这看起来有点麻烦，但边界更清楚。

旧运行时的数据归旧运行时，新运行时的数据归新运行时。尤其是涉及登录态和本地会话时，我更愿意选择一次清晰的重新登录，而不是为了“无感迁移”去碰一堆历史数据。

桌面工具经常会在这里犯错：为了让迁移体验看起来顺滑，把数据边界搞得越来越模糊。

我不想这样做。

## 小工具也需要工程纪律

Codex Quota 的代码量不算大，但我还是给几个核心逻辑写了测试：

- 用量响应解析
- 剩余额度计算
- 状态分类
- 重置时间格式化
- 用量变化对比
- 调试 JSON 脱敏
- 分享摘要不包含账号标识

这些测试不访问 ChatGPT，也不需要真实账号。

它们覆盖的是这个项目最容易出错的地方：数据解析、状态判断、时间展示和隐私边界。

小工具最容易被写成“能跑就行”。但越是贴近日常流程的小工具，越应该避免含糊。它可以功能少，但关键路径要可靠；它可以界面简单，但错误状态要明确；它可以依赖非公开接口，但不能在接口变化时装作一切正常。

## 开源

项目已经开源在 GitHub 上：

- **GitHub 仓库**: [Sanjeever/codex-quota](https://github.com/Sanjeever/codex-quota)

它不是 OpenAI 或 Codex 的官方项目，也没有使用 OpenAI 或 Codex 的官方 logo。使用前请以仓库的源码、Release 与项目说明为准，并自行判断是否接受依赖内部接口的维护风险。

如果你也经常使用 Codex，并且希望把用量状态放在一个更顺手的位置，可以试试看。

对我来说，Codex Quota 的目标一直很简单：把一个需要反复打开网页确认的状态，变成桌面上一眼能看到的信息。

少一点切换，少一点打断。

这就是它的全部价值。
